## Adapter Strategy

### Hybrid: GraphQL Primary + RabbitMQ Secondary

Unlike ERPI (purely passive RabbitMQ), CMMS is best served by a **hybrid adapter**:

1. **GraphQL API** (primary data source) — full CRUD access to all 11 entities with filtering, pagination, and relation inclusion. This is how Forge gets complete snapshots and backfill data.
2. **RabbitMQ subscription** (real-time events) — only `item` topic is currently active. Forge adapter binds its own durable queues to existing fanout exchanges for real-time change notification.
3. **REST API** (fallback/alternative) — same data as GraphQL, simpler to query for single-entity lookups.

### Why Hybrid?

CMMS generates most of its valuable data (work orders, work requests, asset updates) through its **UI** — users create and modify records through the application. These changes are written directly to PostgreSQL via Prisma and exposed through GraphQL/REST. Unlike ERPI where RabbitMQ carries all bidirectional sync traffic, CMMS's RabbitMQ integration is primarily for *consuming* shared master data (items, vendors) from ERPI, not for *publishing* its own domain events.

The Forge adapter should:
- **Poll GraphQL** for work orders, work requests, assets, kits, investigations (CMMS-native data)
- **Subscribe RabbitMQ** for item/vendor/inventory updates (shared master data flowing from ERPI)
- **Expose discovery** of both GraphQL entity types and RabbitMQ topic subscriptions

## Key Observations

1. **Amplication-generated**: Base CRUD is auto-generated; custom logic is in service overrides. The GraphQL schema is a reliable contract.
2. **Asset hierarchy**: The parent/children self-relation on Asset is critical for Forge — it maps to ISA-95 equipment hierarchy (site → area → unit → equipment).
3. **Work order lifecycle**: Rich status tracking with multiple approval checkboxes and safety verifications — this is valuable operational context for decision-quality.
4. **Preventive maintenance scheduling**: cron expressions + period/frequency on WorkRequest/WorkOrder enable predictive maintenance data flows.
5. **globalId federation**: Item and InventoryLocation already have globalId CUIDs — cross-system identity is partially solved.
6. **Inventory backflush on close-out**: WorkOrderService deducts item quantities when work order status becomes "Closed Out" — this is a side-effect Forge should observe.
